# Template 03: Documents you can provide

**Who fills it in:** anyone with access to the system's documentation.
**How:** mark with `[x]` what you have, indicate who holds it and attach it (or tell us
it exists). If you have nothing documented, that is fine: templates 01 and 02 are enough
to start.

> **Before attaching anything:** check there is no data on real people. If there is,
> anonymize the document or do not send it and describe it in words.

---

## What we use each document for

| Document | What it gives us |
|---|---|
| Screenshots | How the result is presented and how the override is captured (what most conditions the decision) |
| Technical spec / model card | Capabilities, limits, and performance we later check are communicated to the user |
| Onboarding / manual | Whether expectations are set well from the start |
| Design / spec | The design intent, to contrast it with the reality of use |
| Usage protocol | The real context where the decision is made |
| Validation / subgroups | Whether limits and possible disparities are known and watched |
| Usage logs (aggregated) | Signals of over-trust, alert fatigue, override rate |
